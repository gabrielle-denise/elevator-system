02 / 11Motivation

Why an elevator is worth proving

01Safety-critical

  • Moves while the door is open → entrapment or injury
  • Motion starts when already at the target → spurious travel
  • Undefined states from incomplete logic → unpredictable behavior

02Testing falls short

  • Simulation explores only a finite subset of behaviors
  • Cannot rule out edge cases by construction

03Proof, not testing

  • A mathematical proof valid for all inputs and states
  • Rocq checks every step against the axioms of type theory
  • Prior work — SPIN, TLA+, B-Method — is bounded; theorem proving is not

Five floors

F0F1F2F3F4

Five commands  δ : State × Command → State

  • CallElevator f — request floor f
  • OpenDoor / CloseDoor — control door when stationary
  • MoveElevator — travel toward the request if safe
  • ArriveAtDestination — complete a journey atomically
04 / 11Transition Function

How the Controller Responds to Commands

δ(s, MoveElevator) — the central case

δ(s, MoveElevator) = s if m(s) = Moving s if d(s) = DoorOpen (f_cur, Closed, Moving, Up, f_req) if f_cur < f_req (f_cur, Closed, Moving, Down, f_req) if f_req < f_cur (f_cur, Closed, Idle, None, f_req) if f_cur = f_req

δ(s, ArriveAtDestination)

(f_req, Closed, Idle, None, f_req) if m(s) = Moving s if m(s) = Idle

Move while door open, open/close while moving, call while moving — all return s unchanged. Safety is enforced structurally, not by runtime assertions.

05 / 11Well-Formedness Invariant

3 Conditions Every Valid State Must Satisfy

1  Floors in bounds

f_cur(s) ∈ [0, 4] ∧ f_req(s) ∈ [0, 4]

2  Door interlock · core

m(s) = Moving ⟹ d(s) = DoorClosed

3  Directional consistency

Idle ∧ dir = None or Moving ∧ Up ∧ f_cur < f_req or Moving ∧ Down ∧ f_req < f_cur
Rules out impossible states → Moving + DirNone Moving + DoorOpen Idle + DirUp / DirDown

07 / 11Assumptions & Axioms

What we assume — and what we leave out

Model assumptions

  • Single pending request — one target, no queue
  • Abstract motion — moves are atomic, no intermediate floors
  • Command-driven — no assumption on command order
  • No emergency stop — overrides not modeled

Axioms used

  • Nat.ltb_irrefl — ¬(n < n)
  • Nat.ltb_lt — ltb ⟺ <
  • lia — closes arithmetic goals automatically

Not proven — by design

  • Liveness — "eventually reaches any requested floor"
  • Requires fairness on the command stream
  • A hostile env could send OpenDoor forever
  • Would need temporal logic or coinductive traces
08 / 11Main Theorems

Six Properties as Formally Stated Rocq Theorems

T1 Door Safety

∀ s, c. inv(s) ∧ δ(s,c).status = Moving ⟹ δ(s,c).door = Closed

T2 Movement Safety

∀ s. inv(s) ∧ f_cur(s) = f_req(s) ⟹ δ(s,Move).status ≠ Moving

T3 State Consistency

∀ s, c. inv(s) ⟹ inv(δ(s, c))

T4 Determinism

s —c→ s₁ ∧ s —c→ s₂ ⟹ s₁ = s₂

T5 Totality

∀ s, c. ∃ s'. s —c→ s'

T6 Reachability + corollaries

∀ s. reachable(s) ⟹ inv(s) — so reachable + Moving: door = Closed f_cur ≠ f_req
09 / 11Proof Sketches

Key Lemmas and Proof Strategies for Each Property

Door Safety ~25 ln

  • Helper lemma: inv ∧ Moving ⟹ door = Closed
  • Apply T3 (preservation) to the successor
  • Apply the helper to that successor state

Movement Safety ~35 ln

  • Case split on movement & door status
  • Already Moving: stutter + strict inequality contradicts f_cur=f_req via lia
  • Idle+Closed: ltb_irrefl builds the Idle state directly

State Consistency ~90 ln

Three canonical shape lemmas:

inv_idle : ∀ f d r. inv(f,d,Idle,None,r) inv_up : f<r ⟹ inv(f,Closed,Up,r) inv_down : r<f ⟹ inv(f,Closed,Down,r)

Reachability ~50 ln

  • Base: initial = (F0,Closed,Idle,None,F0) → inv_idle
  • Step: IH gives inv(s); T3 gives inv(δ(s,c))
  • Structural induction on reachable

Key design insights

  • Inductive Floor type makes invalid floors unrepresentable
  • Stuttering makes δ total and safety structural
  • state_invariant is both the spec and the induction hypothesis

Scope limits — deliberate

  • No request queue, no continuous motion, no emergency stop
  • No liveness — a principled choice, not an oversight
11 / 11Conclusion

What Was Proven, What Was Not, and Where to Go Next

What we built

  • A formal Rocq model of a 5-floor elevator DFSM
  • Machine-checked proofs of 6 properties
  • Covers door interlock, no-op movement, directional consistency

What it demonstrates

  • Theorem proving is practical for safety-critical controllers
  • Proofs are unconditional — not bounded by state space
  • The artifact extracts to certified OCaml code

Future directions

  • Floor-request queue with a scheduling policy
  • Liveness via TLA with fairness assumptions
  • Compare with model checking (TLA+, NuSMV)
  • Extract a certified OCaml controller
